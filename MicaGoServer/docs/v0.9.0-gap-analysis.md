# v0.9.0 Gap Analysis — Contract vs. Implementation

Comparison of [`spec-v0.9.0-client-api-contract.md`](spec-v0.9.0-client-api-contract.md)
against the current implementation (`internal/httpapi/`, `internal/store/`,
`internal/realtime/`, `internal/notify/`) as of this commit.

**Headline:** v0.9.0 is a documentation/stability milestone. The contract was
written to describe what already ships, so **all endpoints, all WebSocket
events, and all model fields in the contract exist today.** The only real gaps
are in *notification provider delivery* (stubs) and a couple of minor metadata
details. No new client-facing routes are required to satisfy the contract.

## REST endpoints

### Fully implemented
All routes are wired in [`router.go`](../micago-server/internal/httpapi/router.go)
with working handlers:

- `GET /api/health`
- `GET /api/server/info`
- `POST /api/auth/check`
- `GET /api/messages/recent`
- `GET /api/chats`
- `GET /api/chats/{guid}/messages`
- `POST /api/chats/{guid}/send`
- `GET /api/attachments/{guid}`
- `POST /api/devices/register`
- `GET /api/devices`
- `PATCH /api/devices/{id}`
- `POST /api/devices/{id}/heartbeat`
- `DELETE /api/devices/{id}`
- `POST /api/devices/{id}/test-push`

### Partially implemented
- `POST /api/devices/{id}/test-push` — endpoint and dispatch logic are complete,
  but actual delivery depends on the provider (see Notification providers
  below). For stub providers it correctly returns `501 not_implemented`.

### Specified but not implemented
- **None.** Every endpoint in the contract's Appendix B exists.

## WebSocket events

### Implemented
Broadcast from `app.go` (sync loop) and `handlers.go` (send flow):

- `message:new` — emitted per newly synced relay row
- `send:match` — emitted when a pending send is confirmed
- `send:error` — emitted on AppleScript failure / send timeout
- `sync:error` — emitted on a failed periodic sync

### Specified but not implemented
- **None.** All four contract events are emitted today.
- Out of scope by design (not in the contract): typing indicators, read
  receipts, reactions/edits/unsend events, replay/backfill over the socket.

## Notification providers

Registered in [`dispatcher.go`](../micago-server/internal/notify/dispatcher.go):

| Provider | Status |
| --- | --- |
| `none` | Implemented (no-op) |
| `webhook` | Implemented (POSTs the notification payload) |
| `fcm` | **Stub** — returns `not_implemented` |
| `hms` | **Stub** — returns `not_implemented` |
| `harmony_push` | **Stub** — returns `not_implemented` (aliased to the HMS stub) |
| `ntfy` | **Stub** — returns `not_implemented` |

All six are accepted at device registration (validation), but only `none` and
`webhook` deliver. Implementing `fcm` / `hms` / `harmony_push` / `ntfy` is
future work and does **not** block the v0.9.0 contract.

## Model fields

### Exist today
Every field in the contract is present in
[`store/models.go`](../micago-server/internal/store/models.go) and
[`notify/payload.go`](../micago-server/internal/notify/payload.go):

- `Health.ok`
- `ServerInfo`: `name`, `version`, `baseUrl`, `websocketUrl`,
  `features.{chats,messages,sendText,attachments,websocket,devices,notifications}`,
  `notificationProviders`
- `Handle`: `id`, `service`
- `Attachment`: `guid`, `filename`, `mimeType`, `transferName`, `totalBytes`,
  `downloadUrl`
- `Chat`: `guid`, `chatIdentifier`, `serviceName`, `displayName`, `isArchived`
- `Message`: `guid`, `text`, `subject`, `service`, `dateCreated`, `dateRead`,
  `dateDelivered`, `isFromMe`, `isRead`, `isDelivered`, `handle`,
  `cacheHasAttachments`, `attachments`
- `Device`: `id`, `name`, `platform`, `clientType`, `pushProvider`,
  `pushEnabled`, `pushTokenSet`, `lastSeenAt`, `createdAt`, `updatedAt`
- List/error envelopes (`data`/`meta`, `error.{code,message}`)
- Push payload: `type`, `messageGuid`, `chatGuid`, `title`, `body`,
  `previewMode`, `createdAt`

### Planned / not yet present
- **No planned-only fields.** The model surface is complete.

## Added in v0.9 (for the macOS companion app)

- **`GET /api/server/status`** — read-only runtime status + permission
  diagnostics consumed by the native companion app. See
  [`spec-v0.10.0-mac-companion.md`](spec-v0.10.0-mac-companion.md). It never
  returns the bearer or push tokens. New models in
  [`store/models.go`](../micago-server/internal/store/models.go):
  `ServerStatusResponse` and its nested types, plus `PermissionCheck`.
- Tests: `TestGetServerStatus`, `TestGetServerStatusTokenNeverExposed`.

The `notifications.implemented` / `notifications.stub` split in the status
payload makes the provider gap (only `none` and `webhook` deliver) visible to
the companion UI.

## Conclusion

The v0.9.0 client contract is fully implemented. The remaining longer-term,
out-of-band task is implementing the stubbed push providers (`fcm`, `hms`,
`harmony_push`, `ntfy`). None of the work here requires BlueBubbles
compatibility, Socket.IO, Firebase, cloud bootstrap, a WebUI/admin page, or
private-API helpers — the control surface is the native macOS companion app.
