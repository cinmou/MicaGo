# v0.11.x Reliability Cross-Check (BlueBubbles source)

Focused source-level comparison of MicaGoServer's slice-2 update pass against the
BlueBubbles reliability paths, to confirm the slice-3 design before implementing.
**Not a broad audit** — only the send/sync reliability areas were re-read.

BlueBubbles files inspected (under `bluebubbles server/packages/server/src/server/`):
- `managers/outgoingMessageManager/messagePromise.ts` (pending match + timeout)
- `managers/outgoingMessageManager/index.ts` (resolve/reject by `isSame`)
- `databases/imessage/pollers/MessagePoller.ts` (`message.error` detection, unsentIds)
- `api/interfaces/messageInterface.ts` (`sendMessageSync`)
- `fileSystem/index.ts` + `api/apple/scripts.ts` (`startMessages`)
- `services/caffeinateService/index.ts` (keep-awake)

## What we just implemented in slice 2

- Bounded **lookback update pass** (default `sync.update_lookback=168h`, `0`
  disables) layered on the rowid-only insert; detects read/delivered/edited/
  unsent/send-error changes on already-tracked messages.
- Persisted **event-state cache** (`message_state` table; per-message
  fingerprint gated by `SchemaCapabilities`) so unchanged rows do not
  rebroadcast; **first sight seeds without broadcasting**.
- Mica-native WebSocket events `message:update` (`changed[]`) and
  `message:unsend`; all detection is capability-gated and degrades gracefully.

## What BlueBubbles does similarly

- **Lookback + event-state cache**: `MessagePoller.poll(after)` looks back **1
  week** on the indexed `date` column, then filters in memory; an `IMessageCache`
  dedupes and detects `updated-entry` via `dateEdited`/`dateRetracted`. This is
  the same shape as our update pass + fingerprint cache.
- **Change set**: it tracks read (`isFromMe`, non-group), delivered (`isFromMe`),
  edits (`dateEdited`), unsends (`dateRetracted`/empty/`hasUnsentParts`) — the
  same fields we fingerprint.
- **Oldest-first processing** and "don't re-emit unchanged" match our pass.
- **Pending send match**: `MessagePromise.isSame` = chatGuid (+ area-code
  variants) + alphanumeric text equality + `sentAt <= dateCreated`. Our
  `FindOutgoingMessageMatch` (normalized text + `sentAt`) is equivalent.
- **Duplicate-send protection**: a `sendCache` keyed by `tempGuid`. Ours is the
  pending manager rejecting a duplicate `tempGuid` with `409` — equivalent.

## What BlueBubbles does differently

- **Send-error detection** lives in the **poller**, not the send call:
  `handlePreviouslyUnsent` partitions rows into `error>0` (errored),
  `isSent && error==0` (sent), `!isSent` (still pending), and on an errored row
  calls `messageManager.reject("message-send-error", entry)`. Because the
  synchronous `sendMessageSync` awaits that same promise, an errored row makes
  the HTTP send **fail fast** (before the 2-minute timeout) and emits a
  `message-send-error` carrying the `tempGuid`. **We do not read `message.error`
  yet** — we only fail on AppleScript error or the 120 s timeout.
- **Outgoing watch-list**: `unsentIds` is carried across polls to follow a sent
  message until it becomes sent/errored. We don't track this across ticks.
- **Timeouts**: 2 min (message) / 20 min (attachment). We use a single 120 s
  text timeout (attachment send is out of scope).
- **Group/system events** (`item_type`/`group_action_type`) — emitted by
  BlueBubbles; we deferred these.
- **Transport/runtime**: file-watcher listeners + pollers; ensures Messages.app
  is running (`startMessages` AppleScript) before AppleScript send; spawns
  `caffeinate -i -m -s -w <pid>` to keep the Mac awake. We use a periodic relay
  sync loop and currently assume Messages.app is open and the Mac is awake.

## What we should implement in slice 3 (design confirmed)

**Send-error fast-fail via `message.error`**, gated by
`capabilities.schema.sendError`:

- During the existing send-confirmation wait, in addition to looking for the
  success match, look for a **matching outgoing row whose `error != 0`** in
  `chat.db`. If found, emit `send:error` (new code `send_error`, carrying the
  numeric Apple error) with the request's `tempGuid` and return immediately,
  instead of waiting out the 120 s timeout.
- Read `message.error` from **chat.db** directly (the relay schema has no error
  column), via a dedicated finder so it works in both `relaydb` and `chatdb`
  api-store modes. Only invoked when `caps.SendError` is true, so the `error`
  column is never referenced on schemas that lack it.
- **Keep** the AppleScript send path, the pending-match manager, and the
  duplicate-`tempGuid` `409`. No private API, no attachment-send queue, no
  Firebase.

This mirrors BlueBubbles' behavior (error → reject pending → fast-fail with
tempGuid) without copying its poller/Socket.IO machinery.

## What we should defer

- A persistent **outgoing watch-list** (`unsentIds` analog) for async
  `send:error` on sends whose error appears after the HTTP call returned. The
  update pass already surfaces `sendError` changes for tracked rows; a dedicated
  watch-list can come later if needed.
- **Group/system events** (`chat:event`) — later slice, behind the
  `groupActions` capability.
- **Auto-retry** of failed sends — BlueBubbles does not meaningfully auto-retry
  either; leave retries to the client.

## What we should not copy

- Private-API send (typing/reactions/edit/unsend send), dylib injection.
- Socket.IO `message-send-error` event shape — keep our Mica `send:error`.
- Attachment-send queue / file staging.
- File-watcher listener stack — our sync loop + update pass is sufficient.
- `caffeinate`/keep-awake and Messages.app keep-alive **inside the Go relay
  core** — these belong in the SwiftUI companion (a later companion slice). The
  Go server may, at most, *detect* and return a clear error when Messages.app is
  not running (slice 4).
