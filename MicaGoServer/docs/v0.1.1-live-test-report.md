## Actual API Behavior Summary

After restarting the current source build:

- `GET /api/chats` defaulted to iMessage-only results
- `GET /api/chats?service=all` returned mixed services
- `GET /api/chats?service=SMS` returned SMS chats only
- `GET /api/chats?service=RCS` returned RCS chats only
- `GET /api/messages/recent` defaulted to iMessage-only and excluded empty non-attachment messages
- `GET /api/messages/recent?service=all&includeEmpty=true` included empty messages
- invalid `service` returned `400`
- invalid `includeEmpty` returned `400`
- invalid `limit` / `offset` returned `400`
- unknown chat GUID returned `404`
- `GET /api/chats/{guid}/messages` matched the expected chat and defaulted to `includeEmpty=false`

## Comparison Against sqlite Output

SQLite inspection summary:

- chat counts by `service_name` showed a mixed dataset with `SMS`, `RCS`, and `iMessage`
- the newest chats overall were mostly `SMS`, while the newest iMessage-only chats were a separate subset
- the newest raw messages included empty iMessage rows
- the newest joined iMessage messages with `(text IS NOT NULL OR cache_has_attachments = 1)` excluded those empty rows

API comparison summary:

- `/api/chats` default matched the iMessage-only subset rather than the newest overall chats
- `/api/chats?service=all` matched the newest overall chats
- `/api/messages/recent` default matched the joined iMessage subset with the non-empty-or-attachment condition
- `/api/messages/recent?service=all` no longer filtered by a literal `"all"`
- `/api/messages/recent?service=all&includeEmpty=true` surfaced the newest empty rows that were excluded from the default view
- `/api/chats/{guid}/messages` matched the targeted sqlite comparison for the same real chat GUID

## Bugs Found

### 1. Stale process on port 3000

The first smoke results were misleading because `127.0.0.1:3000` was already occupied by an older `micago` binary built from a temporary Go build cache path. That instance did not reflect the current source tree.

Impact:

- default chat filtering appeared broken
- empty-message filtering appeared broken
- timestamps appeared wrong even before verifying the current source

Resolution:

- identified the listener with `lsof`
- confirmed it with `ps`
- stopped it
- restarted the current source build from `/Users/Cinmou/Documents/GitHub/MicaGoServer/micago-server`

### 2. Timestamp unit mismatch in live `chat.db`

The current database stores message time values at nanosecond scale relative to the Apple epoch, while the existing helper assumed microseconds only.

Impact:

- `dateCreated`
- `dateRead`
- `dateDelivered`

were returned as inflated future-looking Unix millisecond values.

Resolution:

- updated `internal/timeutil/apple_time.go` to detect large Apple-epoch values and interpret them as nanoseconds
- kept microsecond handling for smaller values
- added `internal/timeutil/apple_time_test.go` to cover both microsecond and nanosecond inputs

- `service=all` for recent messages can still legitimately return only iMessage rows for small limits if the newest matching rows in the real database happen to all be iMessage. This was observed in live data and is not itself a bug.
